\section{Introduction}\label{introduction}

I did not really know how achievable investigator independence in
clinical data management was until I deployed ZZedc on a personal AWS
instance and ran a pilot study without vendor involvement. For years, my
research programmes depended on commercial EDC systems with annual
licensing fees, proprietary data formats, and vendor-controlled
infrastructure. When a vendor relationship deteriorated, the cost of
migration was staggering.

The experience taught me that the barriers to investigator-managed EDC
are not primarily technical. ZZedc installs in minutes, runs on standard
infrastructure, and stores data in portable SQLite databases. The
barriers are organisational: convincing yourself and your institution
that you can manage the infrastructure, security, and maintenance that
vendors handle on your behalf.

This post examines ZZedc as an open-source EDC system designed for
investigator-controlled deployment. It presents three deployment paths
(solo researcher, team research, enterprise), evaluates the technical
architecture, and provides step-by-step guides for each scenario. The
assessment is honest about both the advantages and the significant
responsibilities that come with managing your own clinical data
infrastructure.

\subsection{Motivations}\label{motivations}

\begin{itemize}
\tightlist
\item
  My research programme was paying \$40,000 per year in EDC licensing
  for a system that restricted customisation and locked data in a
  proprietary format.
\item
  A vendor relationship deteriorated, and the estimated cost of data
  migration was \$20,000 -- for data I had collected.
\item
  I wanted to integrate EDC data directly with my R analysis pipelines,
  but the commercial system's export format required manual
  reformatting.
\item
  Multi-site collaborators needed web-accessible data entry, but each
  site's IT department had different policies about vendor-managed
  systems.
\item
  I was curious whether the R/Shiny ecosystem was mature enough to
  support a production EDC application with proper security and
  compliance features.
\end{itemize}

\section{What is Investigator-Controlled
EDC?}\label{what-is-investigator-controlled-edc}

Investigator-controlled EDC means that the research team, rather than a
vendor, manages the electronic data capture infrastructure. The team
controls where data is stored, how it is accessed, and what happens to
it at the end of the study. This contrasts with vendor-managed systems
where data resides on vendor infrastructure under vendor terms.

Think of it as the difference between renting and owning. With a
vendor-managed system, you rent access to a data management platform: it
is maintained for you, but you do not control the infrastructure, and
your data is stored according to the vendor's policies. With
investigator-managed EDC, you own the infrastructure: you control every
aspect, but you are responsible for maintenance, security, and upkeep.

The decision between these approaches is not primarily technical. It
depends on your team's technical capacity, your institution's compliance
requirements, and the balance between licensing costs and staff time for
self-management.

Key design points:

\begin{itemize}
\tightlist
\item
  All infrastructure is standard: Docker containers, standard web server
  (Caddy), open-source database (SQLite).
\item
  No proprietary components: the system uses only open-source software.
\item
  Transparent processes: source code is available for review and
  modification.
\item
  Portable data: SQLite databases can be accessed with any SQL tool.
\end{itemize}

\subsection{What ZZedc Includes}\label{what-zzedc-includes}

\begin{itemize}
\tightlist
\item
  \textbf{Electronic data capture with real-time validation:} Enter
  study data with immediate field-level validation.
\item
  \textbf{Role-based access control:} Five built-in roles (Admin, PI,
  Coordinator, Data Manager, Monitor) with configurable permissions.
\item
  \textbf{Reporting:} Enrollment reports, quality control summaries, and
  statistical overviews.
\item
  \textbf{Data quality framework:} Automated checks for missing data,
  outliers, and consistency.
\item
  \textbf{Audit trail:} Every action is logged with user, timestamp, and
  change history.
\item
  \textbf{Data export:} CSV, Excel, SPSS, and R formats supported.
\end{itemize}

\section{Security and Compliance}\label{security-and-compliance}

ZZedc includes several security features relevant to regulatory
compliance. Users remain responsible for validating that implementations
meet their institutional requirements.

\textbf{Data governance features:}

\begin{itemize}
\tightlist
\item
  User access logging with timestamps and action records.
\item
  Role-based access control with granular permission assignment.
\item
  Data export capabilities supporting data subject access requests.
\item
  Password encryption and session management.
\end{itemize}

\textbf{Technical security:}

\begin{itemize}
\tightlist
\item
  HTTPS support with automatic certificate management via Let's Encrypt.
\item
  Database encryption options.
\item
  Session timeout controls to prevent unauthorised access.
\end{itemize}

\textbf{Important note:} Regulatory compliance (GDPR, 21 CFR Part 11)
depends on implementation context, institutional policies, and
validation within your specific regulatory environment. Coordinate with
your institutional compliance and IT teams.

\subsection{Things to Watch Out For}\label{things-to-watch-out-for}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \textbf{Infrastructure management is real work.} The licensing savings
  from self-managed EDC are offset by staff time for deployment,
  updates, security patching, and troubleshooting. Budget 5-10 hours per
  month for ongoing administration.
\item
  \textbf{Regulatory compliance is your responsibility.} Commercial
  vendors provide compliance frameworks; with self-managed EDC, you must
  validate compliance with your own institutional and regulatory
  requirements.
\item
  \textbf{SQLite has concurrency limitations.} For studies with many
  simultaneous data entry users, SQLite's single-writer constraint may
  cause performance issues. The enterprise path addresses this with
  PostgreSQL.
\item
  \textbf{Domain and DNS configuration adds complexity.} The AWS
  deployment requires a registered domain and DNS configuration, which
  involves coordination with domain registrars and propagation delays.
\item
  \textbf{Backup verification is essential.} Automated backups are only
  useful if you regularly verify that they can be restored. Schedule
  quarterly restore tests.
\end{enumerate}

\subsection{Lessons Learnt}\label{lessons-learnt}

\subsubsection{Conceptual Understanding}\label{conceptual-understanding}

\begin{itemize}
\tightlist
\item
  The barriers to investigator-managed EDC are primarily organisational,
  not technical. ZZedc installs and runs with minimal technical effort;
  the challenge is accepting responsibility for infrastructure.
\item
  Total cost of ownership includes both infrastructure costs and staff
  time. The financial comparison with vendor-managed systems depends on
  local staff costs versus vendor licensing.
\item
  Data portability is a more valuable feature than it appears initially.
  When a vendor relationship ends, portable data formats (SQLite, CSV)
  prevent costly migration projects.
\item
  Open-source EDC requires a different support model: community forums
  and documentation replace vendor helpdesks.
\end{itemize}

\subsubsection{Technical Skills}\label{technical-skills}

\begin{itemize}
\tightlist
\item
  The ZZedc \texttt{init()} function generates a project-specific
  configuration through interactive prompts, requiring no prior R/Shiny
  deployment experience.
\item
  The \texttt{aws\_setup.sh} script automates EC2 instance creation,
  Docker deployment, and Caddy HTTPS configuration.
\item
  SQLite databases can be queried directly from R using the
  \texttt{RSQLite} package, enabling tight integration between data
  capture and analysis.
\item
  Caddy's automatic Let's Encrypt integration eliminates the complexity
  of manual HTTPS certificate management.
\end{itemize}

\subsubsection{Gotchas and Pitfalls}\label{gotchas-and-pitfalls}

\begin{itemize}
\tightlist
\item
  The first deployment takes longer than the documented estimates
  because of DNS propagation delays and AWS account setup.
\item
  SQLite's single-writer limitation is not apparent in testing but
  surfaces under production load with multiple simultaneous users.
\item
  Institutional IT departments may resist investigator-managed
  infrastructure; engage them early in the planning process.
\item
  Security patching is an ongoing responsibility that is easy to
  neglect; schedule monthly maintenance windows.
\end{itemize}

\subsection{Limitations}\label{limitations}

\begin{itemize}
\tightlist
\item
  ZZedc requires technical staff for deployment and maintenance.
  Programmes without IT support capacity should consider vendor-managed
  alternatives.
\item
  The comparison with commercial systems (REDCap, Medidata) is
  incomplete because it does not account for the value of
  vendor-provided training, documentation, and support.
\item
  SQLite is appropriate for small to medium studies but may not scale to
  large multi-site trials with hundreds of concurrent users.
\item
  Regulatory compliance validation must be performed independently for
  each implementation context; ZZedc does not provide pre-validated
  compliance frameworks.
\item
  This post evaluates a single open-source EDC system; alternatives such
  as OpenClinica deserve independent evaluation.
\item
  The author has involvement with ZZedc development, which shapes the
  perspective presented here.
\end{itemize}

\subsection{Opportunities for
Improvement}\label{opportunities-for-improvement}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Develop a comprehensive compliance validation framework that maps
  ZZedc features to 21 CFR Part 11 and GDPR requirements.
\item
  Create a migration toolkit that automates data import from REDCap,
  Medidata, and CSV sources.
\item
  Build a monitoring dashboard that alerts administrators to security
  issues, backup failures, and performance degradation.
\item
  Implement PostgreSQL support for the enterprise deployment path to
  address SQLite's concurrency limitations.
\item
  Develop institutional deployment guides that address common IT
  department concerns about investigator-managed infrastructure.
\item
  Create a cost comparison calculator that accounts for both licensing
  and staff time across different EDC approaches.
\end{enumerate}

\section{Wrapping Up}\label{wrapping-up}

Running ZZedc independently for clinical research data management is
technically achievable and financially attractive for programmes with
the technical capacity to manage their own infrastructure. The system
installs quickly, stores data in portable formats, and provides the core
features -- data entry, validation, audit trails, role-based access --
that clinical research requires.

What I learnt most from this evaluation is that the decision between
investigator-managed and vendor-managed EDC is not about technology. It
is about organisational capacity. If you have staff who can manage AWS
instances, apply security patches, and verify backups, the financial
case for self-managed EDC is strong. If you do not, the hidden costs of
self-management may exceed the visible costs of vendor licensing.

I encourage investigators interested in this approach to start with the
solo researcher path. Deploy ZZedc on your laptop, create a small pilot
study, and evaluate whether the system meets your needs before
committing to AWS deployment. The migration path from local to cloud is
straightforward.

Main takeaways:

\begin{itemize}
\tightlist
\item
  ZZedc provides investigator-controlled EDC with standard data formats
  and open-source code.
\item
  Three deployment paths serve different scales: solo researcher, team
  research, and enterprise.
\item
  Total cost of ownership includes both infrastructure costs and staff
  time.
\item
  Start with a local pilot before committing to cloud deployment.
\end{itemize}
